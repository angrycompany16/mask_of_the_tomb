New game mechanic ideas:
- Buttons... I might have to add buttons
	- Can be triggered either by player stepping on them or being hit by a slambox
	- Can cause various level effects, such as opening doors, changing the level geometry etc. 
- Rotating slamboxes, sort of like tetris pieces. Requires making a 
  function for rotating a rect about its center, but not that much more.
- A mode where the (unnamed) main character controls all of the 
  slamboxes simultaneously, see testslamboxenv.go for an example
  of how this could work! This is a very cool idea.

Bug and improvement list:
- 44100 Hz audio does not work on arch, which is quite a huge problem
- Sometimes spamming pause/play while respawning causes the player to not appear at all
- Opening the game and then exiting from the main menu deletes the current save
- Random crash can occur when loading audio???
- It seems that resetting the game completely causes the game to freeze on playing
  (not sure if it is the 1st or 2nd time) (This is important so please fix)
- Exiting to main menu and then playing the game again causes deadlock
  somewhere
- Loading large sound files mid-game can lead to freezing, so we have to be careful to either
	keep the size of for ex. music files small, or alternatively pre-load them (probably the better)
	option...
- There seems to be a frame perfect bug where if we press the 'Go through	door' button exactly as 
	the player hits the door trigger (?), we may simply phase through the door and disappear...
- It would be very nice if we did not need to call child methods
  each time we inherit from a different class. for example:

  func (t *Transform2D) Update(cmd *engine.Commands) {
	t.Node.Update(cmd)
	...

  It would ideally be automatic to call update on the embedded 
  Node member
- Init is not called when we spawn an actor after the scene has 
  already been initialized. It is only called once a scene is 
  spawned, which means that for newly spawned objects the Camera
  becomes a nil pointer unless we call Init manually...
- Rectify() in SlamboxEnv is currently being called on demand.
  I guess it's fine, but we really shouldn't be doing that kind
  of stuff. It would be better if we just manually re-rectify whenever we actually
	change the level geometry.
